# s03：Permission

本章在 s02 的工具执行前加入确定性的权限管线。安全判断由代码完成，模型不能
自行声明某次调用安全。

## 数据流

```text
FunctionToolCall
    |
    v
1. Hard Deny
    | deny
    v
2. Rule Match
    | 记录是否需要 ask
    v
3. Tool Validation
    | invalid
    v
4. Human Approval
    | ask -> 持久化 checkpoint -> 等待 /approve 或 /deny
    v
ToolRegistry.Execute
```

权限结果有四种：

- `allow`：直接执行。
- `ask`：暂停 Agent，等待人工决定。
- `deny`：硬拒绝，不能通过审批覆盖。
- `invalid`：参数或路径不合法，不进入审批。

`deny` 和 `invalid` 都会转成工具结果回填给模型，模型可以解释失败或改用其他
方案。

## 四层权限管线

### 1. 硬拒绝

`internal/permission` 在其他处理之前检查 bash 调用。s03 永久拒绝：

- `sudo` 等提权命令。
- 关机、重启命令。
- 磁盘格式化和分区命令。
- 向 `/dev` 原始设备写入。
- 递归删除文件系统根目录。
- fork bomb。

`bash` 工具自身也重复执行相同检查。即使调用者绕开 Agent Permission Engine，
硬拒绝仍不会执行。

### 2. 规则判断

默认规则：

- `read_file`、`glob`：自动允许。
- `write_file`、`edit_file`：始终需要审批。
- `bash`：只有确定属于保守只读白名单时自动允许，其他情况全部询问。

Shell 分类刻意保守。`pwd`、`ls`、`cat`、无执行选项的 `find`、部分只读 Git
查询等可以自动运行。`go test`、`awk`、`sed`、Git diff 类命令和任何重定向、
命令替换、命令连接都会询问，因为它们可能执行代码或修改状态。

分类结果只是审批策略，不是 Shell 沙箱。获批 bash 仍可以访问工作区之外的系统
资源。

### 3. 参数校验

每个 `Tool` 新增 `Validate()`：

- 使用严格 JSON 解码，拒绝未知字段。
- 文件工具执行路径和符号链接限制。
- `edit_file` 提前确认精确文本恰好出现一次。
- bash 再次执行硬拒绝检查。

校验发生在询问用户之前，因此非法路径不会生成一个没有意义的审批。

### 4. 人工审批

Agent 遇到第一个 `ask` 后返回：

- 待审批工具名称、参数和原因。
- 已使用的模型 step 数。
- 当前完整 Eino 消息。
- 当前模型响应中的剩余工具调用和位置。

App 将这些内容作为 checkpoint 写入 `.goclaw/approvals/`。审批通过后只执行当前
调用，再继续处理剩余调用和下一次模型请求；拒绝时向模型回填
`Permission denied by user.`。

## Channel 交互

所有 Channel 都支持：

```text
/approve [审批ID]
/deny [审批ID]
/cancel
```

CLI 使用文本审批。飞书实现了可选的 `ApprovalResponder`，发送带“允许”和“拒绝”
按钮的交互卡片。卡片回调转换为相同的 `/approve` 或 `/deny` 消息，因此审批逻辑
仍集中在 App，不依赖飞书类型。

审批绑定原任务的 User ID。群聊中的其他用户以及缺少身份的回调都不能处理审批。

## 重启恢复

等待审批时没有活跃 goroutine。checkpoint 使用 `0600` JSON 文件持久化，进程
重启后：

1. `/status` 显示 `waiting_approval`。
2. 原发起人可以使用原审批 ID 继续。
3. Agent 从持久化消息和 pending call 恢复，不重新请求之前的模型响应。

`/new` 和 `/cancel` 会删除待审批记录。

## 测试

离线测试覆盖：

- 四种权限结果和硬拒绝优先级。
- 保守只读 Shell 分类。
- 工具只校验不执行。
- Agent 审批前暂停、checkpoint JSON 往返、允许和拒绝恢复。
- 写工具审批前不修改文件。
- 等待审批跨 Store 重建恢复。
- 审批发起人身份约束。
- CLI 文本后备和待审批取消。
- 飞书卡片内容和按钮事件转换。
- s01/s02 行为在 s03 规则下保持兼容。

## 当前边界

- 权限规则目前是代码内固定默认值，尚未支持用户或项目配置。
- bash 获批后没有容器、namespace 或系统调用级隔离。
- 只保证“等待审批”状态的重启恢复；审批后执行写操作时发生崩溃，尚未提供
  exactly-once 语义。
- 飞书卡片处理后尚未更新原卡片状态。
- 审计日志和工具生命周期扩展点尚未实现。
