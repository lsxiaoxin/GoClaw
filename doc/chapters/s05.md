# s05：TodoWrite

本章加入会话级 `todo_write` 工具，让模型显式维护当前任务计划。Todo 数据按
chat/session 持久化，工具执行仍走 s02 工具注册表、s03 权限系统和 s04 Hooks。

## 数据流

```text
Channel Message
    |
    v
App attaches chat ID to run context
    |
    v
Agent Loop
    |
    v
todo_write FunctionToolCall
    |
    v
Permission Engine
    |
    v
PreToolUse Hooks
    |
    v
TodoWrite Tool
    |
    v
todo.Store saves .goclaw/todos/<hashed-chat>.json
    |
    v
PostToolUse Hooks
    |
    v
FunctionToolResult returned to model
```

`todo_write` 没有特殊旁路。它注册在 `ToolRegistry` 中，所以参数校验、权限判断、
Hook 阻断和 Hook 注入都沿用现有链路。

## 核心实现

### 1. Todo 模型

`internal/todo` 定义 `Item`、`Status`、`Priority` 和 `Summary`。

支持的状态：

- `pending`
- `in_progress`
- `completed`

支持的优先级：

- `low`
- `medium`
- `high`

校验规则要求 ID 和内容非空、状态和优先级合法、ID 不重复，并且最多只有一个
`in_progress`。这让模型能表达当前正在处理的任务，同时避免多个“当前任务”互相
冲突。

### 2. 持久化 Store

`todo.Store` 按 chat/session ID 保存完整 Todo 列表。文件名使用 chat ID 的 SHA-256
摘要，避免把外部会话 ID 直接暴露成文件名。

默认 CLI 运行时保存到：

```text
.goclaw/todos/
```

目录权限为 `0700`，Todo JSON 文件权限为 `0600`，写入使用临时文件加原子重命名。
不同 chat ID 使用不同文件，不会混在一起。

### 3. TodoWrite 工具

`internal/tool/todo.go` 增加 `todo_write`。它接收当前会话的完整 Todo 列表：

```json
{
  "items": [
    {
      "id": "todo-1",
      "content": "write focused tests",
      "status": "in_progress",
      "priority": "high"
    }
  ]
}
```

工具执行后返回摘要，例如：

```text
Updated 1 todos: 0 pending, 1 in_progress, 0 completed
```

由于 `todo_write` 是普通工具，非法参数会在 Tool/Permission 管线中被拒绝；如果
配置了 Hook，它也会触发 `PreToolUse` 和 `PostToolUse`。

### 4. App 和 Agent 接入

`App` 在启动 Agent Run 前把当前 `ChatID` 注入 context，`TodoWrite` 从 context 中
读取 chat/session ID 后写入对应 Todo 文件。

`/status` 增加 Todo 摘要：

```text
Todo：total=3 pending=1 in_progress=1 completed=1
```

Agent Loop 还会统计连续没有调用 `todo_write` 的模型轮次。连续三轮未更新时，会向
后续模型上下文注入一次提醒，要求模型在多步骤任务中保持 Todo 状态更新。

## 关键文件

- `internal/todo/model.go`：Todo 类型、状态、优先级和校验。
- `internal/todo/store.go`：按 chat/session 持久化 Todo JSON。
- `internal/todo/context.go`：把 ChatID 传入工具执行上下文。
- `internal/tool/todo.go`：`todo_write` 工具实现和参数校验。
- `internal/tool/registry.go`：注册后沿用 Tool/Permission/Hooks 管线。
- `internal/agent/loop.go`：Todo 更新计数和三轮未更新提醒。
- `internal/app/app.go`：Run context 注入 ChatID，`/status` 展示 Todo 摘要。
- `cmd/goclaw/main.go`：创建 Todo Store 并注册 `todo_write`。

## 测试

运行：

```bash
go test ./...
```

s05 的测试覆盖：

- Todo 模型校验和摘要统计。
- 非法状态、非法优先级、重复 ID 和多个 `in_progress`。
- 新增 Todo、更新内容、更新状态、更新优先级。
- 标记 `in_progress` 和 `completed`。
- Todo 持久化保存、重新读取和不同 chat/session 隔离。
- `todo_write` 工具执行和非法参数拒绝。
- `todo_write` 通过现有权限校验和 Hook 注入链路。
- Hook 阻断仍不会执行真实工具。
- `/status` 展示 Todo summary。
- 连续三轮没有更新 Todo 时向模型注入提醒。

## 面试解释

`todo_write` 的关键点不是保存一个列表，而是让 Agent Harness 有一个可观察的任务
状态。模型可以把多步骤任务拆成 `pending`、`in_progress` 和 `completed`，用户和
后续 Prompt 都能看到当前进展。

实现上要强调两点：第一，Todo 按 session 持久化，避免不同会话互相污染；第二，
`todo_write` 是普通工具，不绕过权限和 Hooks。这样新增能力不会破坏前几章建立的
安全边界，也便于后续 s08 上下文压缩和 s10 动态 Prompt 复用 Todo 摘要。
