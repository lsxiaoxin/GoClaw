# s04：Hooks

本章在 s03 权限管线之后加入工具执行前后的 Hook 机制。目标是让 GoClaw 可以在
工具真正执行前拦截或补充上下文，并在工具完成后记录结果或继续向模型注入提示。

## 数据流

```text
FunctionToolCall
    |
    v
Permission Engine
    | deny / invalid -> FunctionToolResult
    | ask            -> approval checkpoint
    | allow
    v
PreToolUse Hooks
    | block -> Hook blocked tool result
    | allow / inject
    v
Tool.Run
    |
    v
PostToolUse Hooks
    |
    v
FunctionToolResult + Hook message
```

Hook 永远不会运行在权限系统之前。硬拒绝、参数非法和等待审批仍由 s03 的权限管线
先处理，Hook 不能把一个被拒绝的工具调用变成可执行调用。

## 核心实现

### 1. HookBus

`internal/hooks` 定义了两个事件：

- `PreToolUse`：工具执行前触发。
- `PostToolUse`：工具执行后触发。

每条 Hook 可以用 `matcher` 匹配工具名：

- 精确工具名，例如 `bash`。
- 通配符 `*`，表示匹配所有工具。

`PreToolUse` 可以放行、阻断或注入消息。阻断时真实工具不会执行，Agent 会收到清晰
的工具结果：

```text
Hook blocked: <reason>
```

`PostToolUse` 可以观察工具输出、错误和耗时，并注入消息。注入消息会追加到同一个
`FunctionToolResult` 中，因此下一轮模型调用能看到这些信息。

### 2. 安全 Runner

s04 提供安全的内置 Runner：

- `allow`：放行。
- `block`：阻断并返回原因。
- `inject`：向工具结果追加提示。
- `record`：当前等价于注入记录信息，后续可接审计日志。

配置中可以声明 `command` 字段，但默认 `BuiltinRunner` 不执行外部命令。外部命令
Hook 保留接口，后续需要明确的信任边界、路径限制和审批策略后再启用。

所有 Hook 都带超时。Hook 超时、返回错误或 panic 不会导致程序 panic；非取消类
失败会变成 Hook 消息回填给模型，原工具继续按既有规则处理。

### 3. 配置加载

默认从工作区读取：

```text
.goclaw/hooks.json
```

也可以通过 `GOCLAW_HOOKS_CONFIG` 指定项目内其他 JSON 文件。相对路径按工作区
解析。

示例：

```json
{
  "hooks": [
    {
      "event": "PreToolUse",
      "matcher": "bash",
      "builtin": "inject",
      "timeout": "500ms",
      "message": "About to run {{tool}} with {{arguments}}"
    },
    {
      "event": "PostToolUse",
      "matcher": "*",
      "builtin": "record",
      "timeout": "500ms",
      "message": "{{tool}} finished in {{elapsed}}"
    }
  ]
}
```

可用模板变量：

- `{{event}}`
- `{{tool}}`
- `{{arguments}}`
- `{{output}}`
- `{{error}}`
- `{{elapsed}}`

## 关键文件

- `internal/hooks/hooks.go`：事件、HookBus、匹配和超时调度。
- `internal/hooks/config.go`：`.goclaw/hooks.json` 解析和校验。
- `internal/hooks/runner.go`：安全内置 Runner 和测试用 FakeRunner。
- `internal/tool/registry.go`：在真实工具执行前后调用 Hook。
- `internal/agent/loop.go`：把 Hook 阻断和注入消息写回模型上下文。
- `internal/config/config.go`：加载 Hook 配置。
- `cmd/goclaw/main.go`：启动时把 HookBus 注入 ToolRegistry。

## 测试

运行：

```bash
go test ./...
```

s04 的测试覆盖：

- Hook 配置解析和非法配置拒绝。
- 精确工具名匹配和 `*` 通配匹配。
- `PreToolUse` 放行、阻断和注入。
- `PostToolUse` 注入。
- Hook 超时、错误和 panic 安全边界。
- Agent 执行工具时调用 Hook。
- Hook 阻断时不继续执行工具。
- Hook 注入信息进入后续模型上下文。
- Hook 不会绕过 s03 权限系统和硬拒绝。
- 审批通过后的工具执行仍会触发 Hook。

## 面试解释

Hooks 是 Agent Harness 的扩展点，不是权限系统的替代品。GoClaw 的顺序是先由
确定性权限管线判断工具是否可以执行，再运行 Hook。这样 Hook 可以做审计、上下文
注入和策略补充，但不能把危险命令绕过硬拒绝或人工审批。

如果面试官追问为什么不直接执行用户脚本，可以说明：外部 Hook 等价于在工具链里
增加一段新代码执行路径，必须先明确脚本来源、工作目录、环境变量、超时、输出上限
和审批模型。s04 先实现内置 Runner 和接口，是为了把生命周期和上下文回填机制做
稳定，再逐步扩展到可信外部命令。
