# s06：Subagent

本章加入同步 `task` 工具，让主 Agent 可以把一个聚焦的调查任务委派给子 Agent。
子 Agent 使用独立消息历史运行，父 Agent 只接收最终摘要，不会把子 Agent 的完整
上下文合并回父上下文。

## 数据流

```text
Parent Agent
    |
    v
task FunctionToolCall
    |
    v
Permission Engine
    |
    v
PreToolUse Hooks
    |
    v
Task Tool
    |
    v
SubagentExecutor creates isolated child Runner
    |
    v
Child Agent with read_file / glob tools
    |
    v
Child final text summary
    |
    v
PostToolUse Hooks
    |
    v
FunctionToolResult returned to parent model
```

`task` 是普通工具，仍然先经过 s03 权限系统，再经过 s04 Hooks。子 Agent 内部调用
工具时，也会走自己的 `ToolRegistry`，因此权限和 Hooks 不会被绕过。

## 核心实现

### 1. Subagent 类型

`internal/subagent` 定义跨包共享类型：

- `Request` / `SubagentRequest`：包含 `prompt` 和可选 `description`。
- `Result` / `SubagentResult`：包含 `status`、`summary` 和可选 `error`。
- `Limits`：包含 `MaxDepth` 和 `MaxConcurrent`。
- `Limiter`：用 channel 控制并发子 Agent 数量。

`Result.Text()` 把子 Agent 结果格式化为父 Agent 可读的工具结果，例如：

```text
Subagent completed: README explains GoClaw.
Subagent blocked by permission: write_file modifies workspace files
Subagent failed: maximum agent steps exceeded
```

### 2. 隔离执行器

`internal/agent/subagent.go` 实现 `SubagentExecutor`。它复用当前模型和 Agent Loop，
但为每次子任务创建新的 `Runner`，并从子任务 prompt 开始构造独立消息列表。

父 Agent 不会拿到子 Agent 的全部消息历史，只会拿到子 Agent 最终输出形成的摘要。
如果子 Agent 失败、取消或请求审批，父 Agent 会收到清晰的错误摘要。

### 3. 限制规则

s06 默认限制：

- 最大递归深度为 `1`，正常子 Agent 不能继续委派 `task`。
- 最大并发为 `2`，通过共享 `Limiter` 控制。
- 子 Agent 默认工具集合为 `read_file` 和 `glob`。

入口 `cmd/goclaw/main.go` 构建两套 registry：

- 父 registry：`bash`、文件工具、`glob`、`todo_write` 和 `task`。
- 子 registry：只包含 `read_file` 和 `glob`，并安装同一份 Hooks 配置。

这样子 Agent 默认只读。若后续扩展危险工具，仍必须通过对应子 registry 的
Permission 和 Hooks 管线。

### 4. Task 工具

`internal/tool/task.go` 新增 `task` 工具，参数示例：

```json
{
  "description": "docs",
  "prompt": "Read README.md and summarize the current project stage."
}
```

工具返回示例：

```text
Subagent completed: README describes GoClaw as a staged coding agent.
```

## 关键文件

- `internal/subagent/types.go`：请求、结果、状态和限制类型。
- `internal/subagent/context.go`：记录当前子 Agent 深度。
- `internal/subagent/limiter.go`：并发限制器。
- `internal/agent/subagent.go`：隔离子 Agent 执行器。
- `internal/tool/task.go`：父 Agent 可调用的同步委派工具。
- `cmd/goclaw/main.go`：父/子 registry 构建和 `task` 注册。
- `internal/agent/loop_test.go`：父子 Agent 集成测试。

## 测试

运行：

```bash
go test ./...
```

s06 的测试覆盖：

- `SubagentRequest` 校验和 `SubagentResult` 文本格式。
- 子 Agent 正常执行并返回摘要。
- 子 Agent 失败时返回错误摘要。
- 子 Agent 请求审批时不自动继续，父 Agent 收到权限摘要。
- 最大递归深度限制。
- 最大并发限制。
- 父 Agent 通过 `task` 调用子 Agent。
- 子 Agent 消息历史不污染父 Agent 上下文。
- 子 Agent 内部工具调用仍经过 Hooks。
- 子 Agent 危险工具调用仍经过 Permission。
- 取消信号会传递到子 Agent。

## 面试解释

Subagent 的价值是把“调查型子任务”从主推理上下文中隔离出去。主 Agent 可以委派
一个聚焦任务，例如“阅读这些文件并总结接口约束”，然后只把摘要带回主上下文。
这样既能控制上下文膨胀，也能让主 Agent 的决策链更清晰。

实现上要强调安全边界：`task` 本身是普通工具，所以父侧不能绕过权限和 Hooks；
子 Agent 也有自己的 registry，默认只给 `read_file` 和 `glob`。深度限制避免递归
委派失控，并发限制避免多个子 Agent 同时挤占资源。
